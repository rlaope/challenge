## 목표 (Goal)

[한컴 닥스(Hancom Docs)](https://www.hancomdocs.com)의 문서를 **읽고 쓸 수 있는 프로그래밍 SDK**를 구축하라.

## 컨텍스트 (Context)

한컴 닥스는 **공개 API가 없는** 한국의 클라우드형 워드 프로세서이다. 에디터는 모든 것을 `<canvas>` 위에 렌더링하므로, **표준 DOM 스크래핑은 동작하지 않는다.** 따라서 웹 에디터의 내부를 **리버스 엔지니어링**하여, **Chrome DevTools Protocol(CDP) 연결을 통해** 문서 콘텐츠를 추출·조작하는 방법을 직접 찾아내야 한다.

> **팁:** 이 챌린지 전반에서 **AI 어시스턴트의 활용이 권장**된다 — 전략 브레인스토밍, 코드 이해, 구현 작성에 모두 사용하라. **AI 도구를 얼마나 효과적으로 활용하는지** 역시 평가 항목이다.

## 요구 사항 (Requirements)

다음을 지원하는, **이미 한컴 닥스 에디터 탭이 열려 있는 Chrome 인스턴스에 연결**하는 라이브러리를 만들어라.

### 필수 기능 (Required Capabilities)

1. **문서 전체 텍스트 읽기** — 문서의 모든 텍스트 콘텐츠 추출.
2. **문서 구조 읽기** — 단락(paragraph), 표(table), 이미지(image)를 별개의 요소로 식별.
3. **서식 읽기** — 임의의 단락에 대해 **폰트 이름·크기, 굵게/기울임 여부, 텍스트 색상**을 알아내기.
4. **텍스트 검색** — 검색어의 모든 출현 위치를 찾고, 매칭별 **문맥(context)** 을 반환.
5. **마크다운(Markdown)으로 내보내기** — 문서를 마크다운으로 변환하되, **헤딩 레벨, 굵게/기울임 마커**를 올바르게 포함.
6. **텍스트 입력(Type text)** — 현재 커서 위치에 텍스트 삽입.
7. **찾아 바꾸기(Find and replace)** — 문서 내 문자열의 모든 출현을 치환.
8. **표(Table) 생성** — 지정된 행·열로 표 삽입.
9. **표 셀 채우기** — 셀 사이를 이동하며 콘텐츠 입력.
10. **문서 저장** — 저장 동작 트리거.

### 보너스 기능 (Bonus Capabilities)

- 특정 **페이지 번호로 이동**.
- **표의 행 삽입/삭제**.
- 단락 스타일 읽기 (**정렬, 줄간격**).
- **이미지 삽입**.
- 모든 **서식 메타데이터를 포함한 구조화된 JSON**으로 전체 문서 내보내기.

## 제약 사항 (Constraints)

- **반드시 Chrome DevTools Protocol(CDP)** 을 사용해야 한다 — Chrome의 **WebSocket 엔드포인트에 연결**하고, 명령을 보내며, **페이지 컨텍스트에서 JavaScript를 평가(evaluate)** 한다.
- 에디터는 모든 것을 **`<canvas>`** 위에 렌더링한다 — **스크래핑 가능한 DOM 텍스트가 없다.**
- `window.getSelection()` 은 빈 값을 반환하고, **클립보드 API는 차단**되어 있다.
- 표준 접근법(DOM 스크래핑, CSS 셀렉터로 텍스트 추출)은 **동작하지 않는다.**
- 문서 콘텐츠를 읽기 위한 **다른 길**을 찾아야 한다.

## 산출물 (Deliverables)

1. **소스 코드** — TypeScript 또는 Python으로 작성된 동작하는 라이브러리:
   - Chrome 및 한컴 닥스 탭에 연결되는 **클라이언트 클래스**
   - 각 필수 기능에 해당하는 **메서드**
   - 최소 1개의 **동작하는 예제 스크립트**
2. **테스트** — 순수 파싱·변환 로직에 대한 **단위 테스트**.
3. **아키텍처 문서** (1페이지 이내) — 다음을 설명:
   - 사용 중인 **API들을 어떻게 발견**했는지.
   - **핵심 기술적 인사이트(key technical insight)** 가 무엇이었는지.
   - 어떤 **한계**가 있는지.
   - **DOM 스크래핑이 안 되는 환경에서** 읽기 접근법이 어떻게 동작하는지.

## 시작하기 (Getting Started)

### 1. Chrome 원격 디버깅 설정

Chrome을 열고 `chrome://inspect/#remote-debugging` 으로 이동. 원격 디버깅을 켜고, **포트 번호를 기록**하라.

### 2. 한컴 닥스 문서 열기

[hancomdocs.com](https://www.hancomdocs.com) 으로 가서 로그인 후, 웹 에디터에서 아무 문서나 연다. URL은 다음과 같은 형태이다:

```
https://webhwp.hancomdocs.com/webhwp/?mode=HWP_EDITOR&docId=...
```

### 3. 에디터 리버스 엔지니어링

웹 에디터는 JavaScript 애플리케이션이다. **그것을 자동화할 수 있을 만큼 동작 원리를 이해하는 것**이 너의 일이다. **어떻게 접근하는지** 자체가 평가 대상이다.

> **유효한 경로는 여러 개**다. 일부는 **부분적 해법**으로 이어지고, **하나는 모든 것을 극적으로 쉽게 만드는 돌파구(breakthrough)** 로 이어진다. 읽기에서 막혔다고 느끼더라도 **계속 탐색하라** — 이 에디터에서 **서식까지 포함된 구조화된 콘텐츠를 가져오는 길**은 분명히 존재한다.

### 4. SDK 빌드

CDP(WebSocket)로 Chrome에 연결하고, 페이지 컨텍스트에서 JavaScript를 평가하며, 키보드 이벤트를 보내라. 라이브러리는 이러한 저수준 동작들을 **깔끔한 고수준 메서드**로 감싸야 한다.

## FAQ

**Q: Puppeteer/Playwright를 써도 되는가?**
A: **Raw CDP를 써라.** 이미 떠 있는 Chrome 인스턴스의 WebSocket 엔드포인트에 연결하고, JavaScript 실행은 `Runtime.evaluate`, 키보드 입력은 `Input.dispatchKeyEvent` 를 사용하라. 이것이 가장 신뢰성 있는 접근법이며, 더 고수준 라이브러리는 **이 에디터가 필요로 하는 작업 위에 불필요한 추상화**를 얹는다.

**Q: 캔버스에서 서식 있는 텍스트를 어떻게 읽을지 도저히 모르겠다면?**
A: **할 수 있는 만큼 구현하라.** **강한 리버스 엔지니어링 사고를 보여주는 부분 해법**도 좋은 점수를 받는다. 그러나 읽기에서 너무 일찍 포기하지는 마라 — **부분 해법과 최선의 해법 사이의 격차는 크고**, 그것을 찾아내는 것이 이 챌린지의 **핵심**이다.
